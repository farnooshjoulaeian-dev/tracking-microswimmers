1. Goal of the notebook
-----------------------

The goal of this notebook is to build a transparent first tracking pipeline for cells detected in a microscopic video.

The pipeline does four main things:

1. Detect cells in each frame using background correction, thresholding, and size filtering.
2. Link detections frame-to-frame to build initial trajectories.
3. Use directional persistence (DP) to penalize abrupt changes in swimming direction.
4. Identify short gaps between broken trajectory segments and merge them only when the jump distance, speed, and DP are physically reasonable.

The notebook keeps the tracking process transparent by storing intermediate tables at each stage of the pipeline:

- detections: all detected cells in every frame before tracking
- tracks: trajectories obtained after initial frame-to-frame linking
- track_summary: summary statistics and endpoints for each trajectory segment
- selected_jmp_links: high-confidence candidate merges between broken trajectory segments
- tracks_merged: final trajectories after applying the selected merges

2. Detection stage
------------------

The detection stage starts by reading the video and extracting basic metadata:

- frame count
- frames per second
- image width
- image height

The function get_video_data(path) uses OpenCV (cv2) to open the video, read its metadata, and return the first frame for testing.

Cell detection is performed by detect_cells_in_frame(...). The function follows the workflow below:

1. Convert the video frame to grayscale using OpenCV (cv2.cvtColor).
2. Estimate a smooth illumination background using a Gaussian filter from SciPy (scipy.ndimage.gaussian_filter).
3. Subtract the grayscale image from the estimated background so that dark cells appear bright.
4. Apply Otsu thresholding using scikit-image (skimage.filters.threshold_otsu).
5. Perform binary closing using SciPy (scipy.ndimage.binary_closing) to connect small fragmented regions.
6. Label connected objects using scikit-image (skimage.measure.label).
7. Measure object properties using scikit-image (skimage.measure.regionprops).
8. Keep only objects whose area falls within the specified size range.

For each accepted object, the notebook stores:

- frame
- label
- centroid position in pixels: x_px, y_px
- centroid position in micrometers: x_um, y_um
- area in pixels and micrometers squared
- mean intensity
- threshold value
- eccentricity
- orientation
- major axis length
- minor axis length

The resulting detections are stored in the detections table.


3. Initial linking strategy
---------------------------

Trajectory reconstruction is performed by linking detections between consecutive frames.

A simple physical estimate provides the maximum expected displacement between frames:

fps = 25
v_cell ≈ 100 μm/s

This gives:

100 / 25 = 4 μm per frame

as the expected displacement of a swimming cell.

In practice, centroid localization noise, image segmentation variability, and small fluctuations in swimming speed can increase the apparent frame-to-frame displacement. To reduce unnecessary track fragmentation, the tracker uses:

max_distance_um = 5.2

This corresponds to:

5.2 μm/frame × 25 frames/s = 130 μm/s

which is slightly above the expected biological speed.

During linking, detections are connected only if they lie within this search radius. Candidate detections are identified efficiently using a cKDTree (scipy.spatial.cKDTree), which avoids comparing every detection with every active track.

When multiple candidates satisfy the distance criterion, they are ranked using a score that combines distance and directional persistence (DP). Distance remains the primary criterion, while DP provides a small preference for candidates that continue the previous direction of motion.

This conservative strategy reduces track fragmentation while minimizing incorrect trajectory connections.


4. Step-alignment score
-----------------------

The function calculate_step_alignment(p0, p1, p2) evaluates how well a candidate detection continues the recent direction of motion of an existing track.

It compares two consecutive displacement vectors:

v1 = p1 - p0
v2 = p2 - p1

and computes:

DP = dot(v1, v2) / (|v1| |v2|)

This quantity is the cosine of the turning angle between the two displacement vectors.

Interpretation:

- DP close to 1 indicates that the candidate continues in nearly the same direction.
- DP close to 0 indicates an approximately perpendicular change in direction.
- DP close to -1 indicates a reversal of direction.

Because the displacement vectors are normalized, DP depends only on directional similarity and is independent of the magnitude of the displacements.

This quantity is not a full velocity autocorrelation function. Instead, it is a local directional-alignment score used to rank competing candidate links during trajectory reconstruction.

The calculation requires three positions:

p0 → p1 → p2

Therefore, DP can only be evaluated once a track contains at least two detections. Newly created tracks initially have no directional history and are linked using distance alone.


5. Frame-to-frame tracking logic
--------------------------------

Tracking begins by assigning a unique track ID to every detection in the first frame.

The video is then processed frame by frame.

For each frame:

1. A cKDTree (scipy.spatial.cKDTree) is built from the detection positions in the current frame.
2. For every active track, the most recent detection is used as the current track endpoint.
3. The cKDTree identifies all candidate detections within the search radius max_distance_um.
4. For each candidate, the following quantities are calculated:
   - track_id
   - detection_a
   - detection_b
   - frame_a
   - frame_b
   - distance_um
   - DP
   - score
5. All candidate links are collected and ranked by score.
6. Candidate links are processed from best to worst score.
7. A candidate link is accepted only if neither the track nor the detection has already been assigned during the current frame.
8. Accepted links extend existing tracks.
9. Any remaining unassigned detections start new tracks.

The candidate score is defined as:

score = distance_um - DP_weight_um × DP

with

DP_weight_um = 0.25

Distance is therefore the primary linking criterion, while DP provides a small preference for candidates that preserve the recent direction of motion.

This weighting is intentionally conservative. DP is used to resolve ambiguities between nearby candidates, but it is not strong enough to override large differences in distance.

Using a cKDTree restricts the search to nearby detections and substantially reduces the computational cost compared with comparing every track against every detection in the frame.

6. Why cKDTree is used
----------------------

The earlier code checked every detection in the current frame against every active track. That becomes slow when there are many detections.

cKDTree speeds this up by only returning detections within the allowed search radius:

nearby_candidate_positions = current_tree.query_ball_point(p1, r=max_distance_um)

This keeps the same scientific rule but reduces runtime.


7. Diagnostic plots after initial linking
-----------------------------------------

The notebook uses several diagnostics after creating tracks:

1. Track length distribution

This shows whether tracks are mostly short or whether many tracks continue for a large part of the video.

For a 10 second video at 25 fps, a full-length track can be around:

10 * 25 = 250 frames

2. Accepted link distances

This shows how close accepted links are to max_distance_um.

If many links pile up near the maximum, the distance threshold is strongly controlling the result.

3. DP distribution

This shows whether accepted links are directionally smooth.

A strong peak near 1 means many links continue the previous direction.

4. Longest-track plots

These are used for visual inspection. They show whether long tracks look smooth, fragmented, or wrongly connected.


8. Current tracking result before merging
-----------------------------------------

The frame-to-frame DP-based tracker produced many usable tracks, including long tracks near the full video length.

However, visual inspection showed that some real trajectories were split into separate track segments.

This happened because the main linker only connects:

frame t -> frame t+1

It does not allow a cell to disappear temporarily and reappear later.

Therefore, a second stage was added: jump detection for broken tracks.


9. Jump detection for broken tracks
-----------------------------------

Jump detection works at the track-segment level, not at the individual detection level.

For each track, the notebook summarizes:

- track_id
- start_frame
- end_frame
- start_x_um
- start_y_um
- end_x_um
- end_y_um
- length

Then it checks whether one track segment can reasonably continue into another.

For a candidate pair:

track A ends at frame t
track B starts at a later frame

The code calculates:

jmp_frames = start_frame_B - end_frame_A
jmp_distance_um = distance from end of A to start of B
jmp_speed_um_s = jmp_distance_um / jmp_frames * fps
jmp_DP = direction consistency from the end of A toward the start of B

The current jump parameters are:

max_jmp_frames = 3
max_jmp_speed_um_s = 130
max_jmp_distance_um = max_jmp_speed_um_s * max_jmp_frames / fps
min_jmp_DP = 0.5
min_segment_length = 20

The dynamic distance rule is important:

max_jmp_distance_um = max_jmp_speed_um_s * max_jmp_frames / fps

This means the allowed jump distance scales with the number of missing frames.

For example, with fps = 25, max_jmp_frames = 3, and max_jmp_speed_um_s = 130:

130 * 3 / 25 = 15.6 um

This is more scientifically consistent than using a fixed jump distance for all missing-frame intervals.


10. Selecting jump links
------------------------

The notebook filters candidate jumps by:

- jump duration
- jump speed
- jump DP
- minimum length of both track segments

The selected links are stored in:

selected_jmp_links

Each selected link says:

track_b should be merged into track_a

where:

- track_a is the earlier segment
- track_b is the later segment

Before merging, selected links should be plotted and inspected visually. This is important because larger jumps can accidentally connect the wrong cells in dense regions.


11. Merging broken tracks
-------------------------

The notebook uses a parent-map method for merging.

Important distinction:

tracks is the original track table.
track_parent is only a dictionary of merge instructions.
tracks_merged is the repaired table after applying those merge instructions.

At first:

track_parent[track_id] = track_id

Each track is its own parent.

Then selected_jmp_links update the map:

track_parent[parent_b] = parent_a

This means:

merge the later track segment into the earlier parent track.

The merge is applied by:

tracks_merged = tracks.copy()
tracks_merged["original_track_id"] = tracks_merged["track_id"]
tracks_merged["track_id"] = tracks_merged["track_id"].apply(lambda track_id: find_parent(int(track_id)))

The original_track_id column is important. It preserves the pre-merge track ID so manual inspection remains possible.

Renumbering is intentionally disabled during debugging. If track IDs are renumbered too early, old IDs disappear and manual inspection becomes confusing.

13. Manual inspection logic
---------------------------

The notebook includes a manual inspection cell for two selected tracks.

This cell checks two cases:

1. Time overlap

If two tracks exist in the same frames, they usually should not be merged as one broken track.

The code calculates:

overlap_frames
overlap_distances

If overlap exists, the distance between tracks during overlapping frames is measured.

2. Jump from A to B

If the tracks do not overlap, the code checks whether track A can jump to track B.

It calculates:

jmp_frames
jmp_distance_um
jmp_speed_um_s
jmp_DP

Decision rule:

- If tracks overlap in time, usually do not merge.
- If they do not overlap, merge only if jump duration, speed, distance, and DP are reasonable.
